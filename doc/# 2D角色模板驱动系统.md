# 2D角色模板驱动系统（Template-driven Character Animation Engine）需求说明书（整理版）

## 1. 背景与定位
本系统面向“单张2D角色图（全身立绘/设定图）→ 可控、可复用的实时动画”的需求。核心思路不是一次性生成动画视频，而是**把输入图转成可运行的角色模型（Rig + 参数系统），再用参数驱动动作**，从而获得稳定、可控、可复用的动画能力。

一句话总结：  
**用 AI 自动完成类似 Live2D Cubism 的“建模 + rigging + 驱动”流程。**

---

## 2. 核心目标（第一阶段）
第一阶段目标不是“AI生成复杂动作/文生视频”，而是做一个可运行的最小动画运行时（Runtime），实现：

- 输入结构化动作/状态（JSON/DSL）
- 角色能稳定地“动起来”（眨眼、呼吸、说话、轻微摆动、情绪切换）
- 可实时控制与切换剧情/镜头/背景
- 可复用 motion（动作资产）并可扩展到新角色

第一阶段最终形态：  
**“AI视觉小说运行时（VN Runtime）/ VTuber Runtime / 2D角色播放器”，但可编程。**

---

## 3. 总体流程（端到端）
```text
用户输入：一张全身图
        ↓
自动结构解析（AI分割 + 识别）
        ↓
映射到角色模板（Template）
        ↓
生成绑定系统（Rig + Params）
        ↓
参数驱动动画（Motion System）
        ↓
渲染输出（实时/录制视频）
```

---

## 4. 核心概念定义

### 4.1 模板不是图片：结构模板（Character Schema）
模板的本质是“可动画的身体结构图谱”，定义角色应包含的部件与层级。

示例（概念）：
```text
HumanTemplate
- head
- eyes
- mouth
- torso
- arms
- legs
- hair
```

模板价值：
- 结构固定（可控、可复用）
- 动作资产可复用（motion 独立于角色图）
- 支持扩展到不同生物类型（人/动物/植物）

### 4.2 参数系统（Params）
系统不是直接产出“动画帧”，而是生成一套可被驱动的参数，例如：
```text
ParamHeadAngleX
ParamHeadAngleY
ParamArmSwing
ParamBreath
ParamBlink
```

参数来源：
- idle motion（待机）
- wind（风摆）
- dance（舞蹈）
- speech（说话/口型）
- emotion（情绪状态）

### 4.3 Rig 与驱动
参数最终作用在角色上，驱动方式可包含：
- rigid transform（刚体位移/旋转/缩放）
- mesh warp / deformation（网格变形）
- skeletal animation（骨骼动画/IK）
- blend/mix（动作混合）

## 6. Demo 目标（第一阶段验收）
界面布局（示意）：
- 左侧：角色显示
- 右侧：剧情/控制台（输入 JSON/DSL）

输入示例：
```json
{
  "character": "girl_01",
  "action": "talk",
  "emotion": "happy",
  "camera": "medium"
}
```

期望表现：
- 角色：微笑、呼吸、眨眼、嘴巴说话、身体轻微摆动
- 镜头：自动拉近/切换
- 背景：可切换

---

## 7. Scene DSL（关键资产）
DSL 是系统壁垒：未来小说脚本/LLM/对话系统最终都转成它，而不是直接生成视频。

示例：
```json
{
  "scene": "classroom",
  "characters": [
    { "id": "girl_01", "action": "talk", "emotion": "happy" }
  ],
  "camera": { "type": "medium" }
}
```

---

## 8. 技术选型（第一阶段推荐）

### 8.1 渲染层：PixiJS
定位：GPU 2D 游戏引擎（WebGL/WebGPU），多平台友好，适合实时控制与 Electron。

角色最终表现形态：
- 一个 Pixi Container
- 内部由身体/头/眼/嘴/特效等子节点组成

### 8.2 角色动画层：Spine（推荐路线）
定位：2D骨骼动画编辑与运行时，适合程序化控制：
- 切换动画：`setAnimation("talk")`
- 直接控制骨骼：`findBone("head").rotation = -12`

第一阶段不建议以 Live2D 为主：
- 参数体系复杂、制作重、自动化弱、Runtime 可控性较差（对“AI控制”不友好）

### 8.3 时间轴/补间：GSAP
用于镜头、UI、呼吸、淡入淡出、摇晃、表情变化等通用时间线控制。

---

## 9. 第一阶段范围（必须约束）

### 9.1 要做（Do）
- PixiJS 跑起来：背景 + 角色显示
- Spine 角色跑起来：idle/talk/blink
- Character 类抽象：角色不等于 Spine（Spine 只是渲染层）
- EmotionController：`setEmotion("sad")` 触发眉眼嘴/呼吸/姿态联动
- Camera 系统：zoom/pan/shake/follow
- Scene DSL：输入 JSON 让角色“自动演”
- Timeline：剧情自动播放（按时间触发动作/镜头/背景/对白）

### 9.2 不做（Non-goals）
- AI 自动动作生成（太早、不可控）
- 文生视频（方向错误）
- Unity/Unreal 重型引擎（开发成本高）
- 自研骨骼系统（投入大且收益低）
- 复杂物理（第一阶段无必要）

---

## 10. 关键难点与风险

1) 自动分割质量  
- mask 错 → rig 错 → 动作全错

2) mapping 不稳定（部件归属错误）  
- 左臂识别成衣服、头发粘脸等，导致模板映射失败

3) 参数设计（最关键）  
- 需要一套通用 motion schema，让 wind/dance/emotion/speech 在同一参数体系下可组合、可复用

---

## 11. 开发顺序（建议）
1) PixiJS：显示背景与角色容器  
2) Spine：角色动作（idle/talk/blink）跑通  
3) Character 抽象：统一 API（action/emotion/state）  
4) Camera：基础镜头语言（拉近/摇晃/跟随）  
5) Scene DSL：输入 JSON 驱动演出  
6) Timeline：剧情自动播放

第一阶段完成后：  
LLM 接入只需“生成 Scene DSL”，不需要生成视频，整体难度显著降低。
