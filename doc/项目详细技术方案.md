# 项目详细技术方案：2D角色模板驱动系统（Template-driven Character Animation Engine）

## 0. 文档目的
本文档给出“2D角色模板驱动系统”的可落地技术方案，覆盖第一阶段（最小可运行 Runtime）到后续阶段（自动分割/自动 Rigging/模板化资产）的演进路径，并明确模块边界、数据协议（Scene DSL）、运行时架构与关键实现要点。

---

## 1. 总体目标与边界

### 1.1 第一阶段目标（必须达成）
- 输入 Scene DSL（JSON）即可驱动演出：角色、动作、情绪、镜头、背景、对白
- 角色稳定可控：idle、talk、blink、breath、轻微摆动、情绪切换（happy/sad/angry）
- 镜头语言可用：zoom、pan、shake、follow、shot preset（wide/medium/close）
- 可复用动作资产：同一套动作可复用到多个角色（至少同骨架或同模板族）
- 可实时交互：控制台/接口输入 DSL，立刻生效（不需要重新生成视频）

### 1.2 第一阶段非目标（明确不做）
- 文生视频/端到端视频生成
- 复杂物理（布料/毛发真实物理）
- 自研骨骼/网格系统
- 自动动作生成（从文本推理动作）

### 1.3 后续阶段目标（规划）
- AI 分割与部件识别：从单张立绘提取结构层
- 模板匹配与映射：分割结果映射到 Character Schema
- 自动 Rig 生成：输出可驱动的角色模型（骨架/网格/参数绑定）
- 通用 Motion Schema：支持 wind/dance/speech/emotion 的组合与复用

## 2. 技术栈与运行形态

### 2.1 推荐技术栈（第一阶段）
- 语言：TypeScript
- 渲染：PixiJS（WebGL/WebGPU）
- 骨骼动画：Spine Runtime for PixiJS
- 时间轴：GSAP
- 构建：Vite
- 运行形态：Web（优先），可扩展 Electron（桌面壳）

### 2.2 关键设计原则
- 数据驱动：一切演出由 DSL 驱动，Runtime 只负责解释与执行
- 分层清晰：Character（AI角色层）与 Spine（渲染层）解耦
- 可扩展：新增动作/情绪/镜头无需改动引擎核心
- 可观测：运行时事件、状态快照、资源加载状态可查询

### 2.3 Spine 的组成与许可边界（必须理解）
Spine 在工程里分成两部分，很多团队会在这里第一次踩坑：

- Spine Editor：骨骼动画制作工具，不开源
- Spine Runtime SDK：播放动画的运行时，源码开放（但不是 MIT 这类完全自由协议）

第一阶段引擎真正依赖的是 Runtime SDK，而不是 Editor。

#### 2.3.1 Web/PixiJS 技术选型
- PixiJS v8 对应：`@esotericsoftware/spine-pixi-v8`
- TypeScript Runtime 对应仓库/目录：Spine Runtimes（含 `spine-ts/`，以及 `spine-ts/spine-pixi-v8/`）

#### 2.3.2 Runtime 工作方式（数据与调用链）
- Editor 导出：`skeleton.json` + `atlas` + `png`
- Runtime 加载并实例化：从 skeleton/atlas 创建 Spine 实例
- 播放动画：例如 `state.setAnimation(trackIndex, name, loop)`

#### 2.3.3 授权与合规提示
- Runtime 源码开放 ≠ 完全自由协议
- Spine Runtime 受其 Runtime License 约束，商业使用需要满足 Spine 生态的授权要求
- 项目启动时应明确：资产制作（Editor）与发布产品的授权闭环，避免后期被许可问题卡住交付

## 3. 运行时总体架构

### 3.1 核心模块
```text
Engine Runtime
├── Renderer（Pixi App + 根容器）
├── Asset（资源加载、缓存、版本）
├── Scene（场景图、角色层、背景层、特效层、UI层）
├── Character（角色抽象：状态、动作、情绪、说话）
├── Animation（动作/混合/参数驱动适配）
├── Emotion（情绪到表情与动作的映射）
├── Camera（镜头：缩放、平移、抖动、跟随、预设）
├── Timeline（剧情时间轴：事件调度、并行/串行、可跳过）
└── DSL（解析、校验、编译为内部指令）
```

### 3.2 数据流（第一阶段）
```text
Scene DSL(JSON)
  ↓ parse/validate
SceneProgram（内部指令序列）
  ↓ schedule
Timeline（按时间触发）
  ↓ commands
Scene/Character/Camera
  ↓ update per frame
Renderer（Pixi）→ 显示
```

### 3.3 时序模型
- 固定主循环：`update(dt)` + `render()`
- Timeline 负责“何时发生什么”
- GSAP 负责“过渡如何发生”（镜头/淡入淡出/缓动）

## 5. Scene DSL 技术规格（第一阶段）

### 5.1 顶层结构（建议）
```json
{
  "version": "1.0",
  "scene": {
    "id": "classroom_day",
    "background": { "assetId": "bg_classroom_day" }
  },
  "characters": [
    {
      "id": "girl_01",
      "prefabId": "girl_01_spine",
      "transform": { "x": 520, "y": 820, "scale": 1.0 },
      "state": { "action": "idle", "emotion": "neutral" }
    }
  ],
  "camera": { "shot": "medium" },
  "timeline": [
    { "t": 0.0, "cmd": "Character.SetAction", "args": { "id": "girl_01", "action": "idle", "loop": true } },
    { "t": 0.0, "cmd": "Character.SetEmotion", "args": { "id": "girl_01", "emotion": "happy" } },
    { "t": 0.2, "cmd": "Dialogue.Say", "args": { "id": "girl_01", "text": "你好。" } },
    { "t": 0.2, "cmd": "Character.Talk", "args": { "id": "girl_01", "enabled": true } },
    { "t": 2.0, "cmd": "Character.Talk", "args": { "id": "girl_01", "enabled": false } },
    { "t": 2.1, "cmd": "Camera.ToShot", "args": { "shot": "close", "duration": 0.6 } }
  ]
}
```

### 5.2 命令集合（最小集）
- Scene.Load：加载场景与资源
- Scene.SetBackground：切换背景（支持过渡）
- Character.Spawn：生成角色
- Character.Despawn：移除角色
- Character.SetTransform：移动/缩放/层级
- Character.SetAction：切换动作（Spine Animation）
- Character.SetEmotion：切换情绪（驱动表情/姿态）
- Character.Talk：开启/关闭说话状态（口型/说话动作混合）
- Camera.ToShot：切换镜头预设（wide/medium/close）
- Camera.PanTo / Camera.ZoomTo / Camera.Shake：镜头效果
- Dialogue.Say：显示对白（可与 Talk 绑定）

### 5.3 校验策略
- 版本号：`version` 必须存在
- 引用校验：`assetId`、`prefabId`、`character.id` 必须可解析
- 时间轴：`t` 非递减；同一时间点可多指令
- 命令参数：按命令 schema 校验（缺参/错类型直接拒绝执行）

## 6. 内部数据模型（建议）

### 6.1 Character（AI角色层）
职责：对上提供“意图 API”（action/emotion/talk），对下驱动 Spine/渲染节点。

建议接口（概念）：
```ts
export type CharacterId = string;

export type ActionName = string;

export type EmotionName =
  | "neutral"
  | "happy"
  | "sad"
  | "angry"
  | "surprised";

export interface CharacterStateSnapshot {
  id: CharacterId;
  action: ActionName;
  emotion: EmotionName;
  talkEnabled: boolean;
  x: number;
  y: number;
  scale: number;
}

export interface Character {
  readonly id: CharacterId;
  spawn(): Promise<void>;
  destroy(): void;

  setTransform(x: number, y: number, scale: number): void;
  setAction(action: ActionName, loop: boolean): void;
  setEmotion(emotion: EmotionName): void;
  setTalkEnabled(enabled: boolean): void;

  getSnapshot(): CharacterStateSnapshot;
  update(dtMs: number): void;
}
```

### 6.2 CharacterView（渲染层适配）
核心观点：Character 不等于 Spine。Spine 只是可替换的 View。

```ts
export interface CharacterView {
  attachTo(container: unknown): void;
  detach(): void;

  setAction(action: string, loop: boolean): void;
  setSkin(skin: string): void;
  setMix(from: string, to: string, duration: number): void;

  setSlotAlpha(slot: string, alpha: number): void;
  setBoneRotation(bone: string, rotationDeg: number): void;
  setBonePosition(bone: string, x: number, y: number): void;

  update(dtMs: number): void;
}
```

### 6.3 EmotionMap（情绪映射）
情绪不是单一动作，而是“多个通道的组合”：表情 slot、眉眼嘴形态、姿态、呼吸幅度、idle 摆动等。

示例（数据化配置）：
```json
{
  "neutral": { "actionOverlay": "idle", "brow": 0.0, "mouth": 0.0, "breath": 0.6 },
  "happy":   { "actionOverlay": "idle", "brow": 0.2, "mouth": 0.8, "breath": 0.7 },
  "sad":     { "actionOverlay": "idle", "brow": -0.2, "mouth": -0.4, "breath": 0.5 },
  "angry":   { "actionOverlay": "idle", "brow": 0.6, "mouth": -0.2, "breath": 0.8 }
}
```

实现策略（第一阶段）：
- 优先使用 Spine 的 slot 切换、skin、或已有表情动画片段
- 如果资源不具备丰富表情，则退化为“切换表情动画 + 叠加轻微骨骼旋转”

## 7. Renderer 与 Scene Graph 方案

### 7.1 Pixi 层级
建议固定分层，保证渲染顺序可控：
```text
Root
├── BackgroundLayer
├── WorldLayer
│   ├── CharactersLayer
│   └── EffectsLayer
└── UiLayer
```

### 7.2 Camera 实现方式
不直接移动 Pixi 摄像机（Pixi 无内置 camera），而是：
- 将 `WorldLayer` 挂到 `CameraContainer`
- Camera 通过变换 `CameraContainer`（scale/position）实现镜头效果

镜头参数：
- `zoom`：等价于 `CameraContainer.scale`
- `pan`：等价于 `CameraContainer.position`
- `shake`：在基准变换基础上叠加小幅噪声/GSAP timeline
- `follow`：基于目标角色屏幕位置做平滑追踪

### 7.3 Shot 预设（wide/medium/close）
Shot 是“约定的构图”，由一组参数定义：
```json
{
  "wide":   { "zoom": 0.85, "offsetX": 0,  "offsetY": 0 },
  "medium": { "zoom": 1.05, "offsetX": 0,  "offsetY": -40 },
  "close":  { "zoom": 1.30, "offsetX": 0,  "offsetY": -80 }
}
```

镜头切换用 GSAP 进行插值，保证平滑过渡。

## 8. 动作系统与混合策略（Spine）

### 8.1 动作分类
- Base Loop：idle、breath、blink（常驻循环）
- Action：talk、walk、wave（显式切换）
- Overlay：姿态微调（骨骼旋转、slot alpha）用于情绪与镜头语言增强

### 8.2 Spine 混合（MixPolicy）
第一阶段建议规则化：
- idle ↔ talk：设置固定 mix（例如 0.15~0.25s）
- talk 开启：切换到 talk loop 或在 track 上叠加 mouth/talk
- blink：尽量做独立小动画，可定时触发或常驻随机触发

### 8.3 Talk（说话）实现
第一阶段提供三档实现路径（按资源条件自动降级）：
1) 资源含嘴型/口型动画：Talk 开启时播放 mouth track 或 talk 动作
2) 资源含 talk 动作但无嘴型：Talk 开启时切换 talk loop
3) 资源仅 idle：Talk 开启时做轻微头部/胸口摆动 + 字幕出现

## 9. Timeline（剧情时间轴）设计

### 9.1 执行模型
- Timeline 输入：已编译的指令列表（按 t 排序）
- 执行时维护：
  - 当前时间 `time`
  - 待执行指令指针 `cursor`
  - 正在运行的过渡（GSAP timeline 列表）

### 9.2 并行与串行
第一阶段建议只支持：
- 同一时间点命令并行执行
- 命令的过渡时间由 args 指定（例如 camera duration）
- Timeline 不阻塞：对白显示不影响后续事件推进（如需要阻塞，后续阶段扩展 wait/await）

### 9.3 跳过与快进
要求：
- 支持 `seek(t)`：直接跳转到某时间点
- seek 时策略：
  - 停止所有 GSAP 过渡
  - 重放从 0 到 t 的“状态类命令”（SetAction/SetEmotion/SetBackground/SetTransform）
  - 忽略一次性命令（如 Shake）或按策略选择是否补偿

### 9.4 命令执行语义

#### 9.4.1 状态命令 vs 一次性命令
用于解决两个核心问题：
- seek 时哪些命令需要重放才能恢复到目标状态
- 回放时哪些命令只执行一次，不参与“状态还原”

状态命令（参与 seek 重放，seek 时以“瞬时设置”方式应用，忽略 duration/缓动）：
- Scene.Load（可选：seek 时不强制重载，仅确保资源已加载）
- Scene.SetBackground
- Character.Spawn / Character.Despawn
- Character.SetTransform
- Character.SetAction
- Character.SetEmotion
- Character.Talk（enabled 属于状态）
- Camera.ToShot（seek 时直接设置到目标 shot 的最终值）
- Camera.PanTo / Camera.ZoomTo（seek 时直接设置到最终位置/缩放）

一次性命令（不参与 seek 重放，seek 时默认忽略）：
- Camera.Shake
- Dialogue.Say（对白展示属于时刻事件，不作为场景状态）
- Effects.Play（第一阶段可选）

可选扩展（第二阶段再做）：对白/特效可标记为 `replayOnSeek: true`，在 seek 后从目标时间点“重放窗口内事件”。

#### 9.4.2 命令失败策略（是否中断 Timeline）
默认策略（推荐，便于演出不中断）：
- 非致命失败：记录事件并继续执行后续命令
- 致命失败：中断 Timeline（例如关键资源缺失导致场景无法继续）

建议引擎提供两级控制：
- 全局模式：`strict`（任意失败中断）/ `lenient`（非致命继续）
- 单命令覆盖：命令可携带 `fatal: true` 强制中断

#### 9.4.3 可重试机制
用于资源抖动或异步加载未就绪场景。建议仅对“可恢复失败”启用重试，例如：
- 资源未加载完成
- 网络/IO 临时失败（Web 环境）

建议命令可携带：
```json
{
  "retry": { "times": 2, "delayMs": 150 }
}
```

#### 9.4.4 失败事件格式（建议）
建议统一事件结构，便于调试面板与日志系统消费：
```json
{
  "type": "CommandFailed",
  "sceneId": "classroom_day",
  "timelineTime": 2.1,
  "cmd": "Camera.ToShot",
  "args": { "shot": "close", "duration": 0.6 },
  "error": {
    "code": "ASSET_NOT_FOUND",
    "message": "prefabId not registered: girl_01_spine",
    "details": { "prefabId": "girl_01_spine" }
  },
  "policy": {
    "fatal": false,
    "retried": 1,
    "willContinue": true
  }
}
```

## 10. 资源与资产管线

### 10.1 资源类型
- Background：图片（jpg/png/webp）
- Character Prefab：Spine 导出（skeleton json/binary + atlas + textures）
- Effects：粒子或序列帧（第一阶段可选）
- UI：字体、气泡、立绘名牌

### 10.2 AssetManager 关键能力
- 统一加载接口：按 `assetId` 加载并缓存
- 资源引用计数：Scene 切换时释放不再使用资源
- 失败策略：加载失败返回可诊断错误（assetId、url、原因）

### 10.3 资源命名约定（建议）
- `bg_*`：背景
- `char_*`：角色 prefab
- `fx_*`：特效
- `ui_*`：UI 资源

### 10.4 角色 Prefab 规范（MUST）
本节用于保证“同一套 Runtime/DSL 在不同角色上稳定可用”，避免角色资源差异导致 Emotion/Talk/MixPolicy 失效。

#### 10.4.1 必需动画（MUST）
- `idle`：待机循环（必须存在）
- `talk` 或 `mouth track`：说话表现（二选一必须满足）
  - 方案 A：存在 `talk` 循环动画（推荐）
  - 方案 B：存在独立口型动画（例如 `mouth_talk`），可在指定 track 上叠加（需要 prefab 配置）

#### 10.4.2 推荐动画（SHOULD）
- `blink`：眨眼（推荐；无则由程序用低质量替代方案兜底）

#### 10.4.3 可选情绪动画（MAY）
- `emotion_*`：例如 `emotion_happy` / `emotion_sad` / `emotion_angry`
  - 若存在：EmotionController 可优先切换到对应动画或做短过渡
  - 若不存在：使用绑定规则（slot/bone 微调 + idle overlay）实现情绪

#### 10.4.4 必需骨骼/Slot 命名约定（MUST）
第一阶段先约束 3–5 个关键点，确保镜头与表情有最小控制面：
- 骨骼（Bones，至少满足其中 3 个）：
  - `head`：头部（用于轻微点头/转向/情绪姿态）
  - `chest`：胸腔/上身（用于呼吸与整体姿态）
  - `root` 或 `hips`：根/髋（用于整体位移或轻微摆动）
- 槽位（Slots，至少满足其中 2 个）：
  - `mouth`：嘴部相关 slot（用于 Talk/表情切换）
  - `eyeL`、`eyeR`：左右眼 slot（用于 blink 或表情）

允许“别名映射”：如果美术资源命名不同，必须在 prefab 配置中显式映射到上述标准名。

#### 10.4.5 Mix 规则（默认值 + 可覆盖）
默认混合规则（引擎内置）：
- `idle ↔ talk`：0.2s
- 其他未配置动作切换：0.15s

按角色覆盖（prefab 配置）：
- 支持为不同动作对定义 mix duration
- 支持设置全局 defaultMix

建议 prefab 描述（示例）：
```json
{
  "prefabId": "girl_01_spine",
  "spine": {
    "skeleton": "assets/char_girl_01/skeleton.json",
    "atlas": "assets/char_girl_01/skeleton.atlas",
    "textures": ["assets/char_girl_01/skeleton.png"]
  },
  "animations": {
    "idle": "idle",
    "talk": "talk",
    "blink": "blink"
  },
  "talk": {
    "mode": "action",
    "action": "talk"
  },
  "bindings": {
    "bones": { "head": "head", "chest": "chest", "root": "root" },
    "slots": { "mouth": "mouth", "eyeL": "eyeL", "eyeR": "eyeR" }
  },
  "mix": {
    "defaultMix": 0.15,
    "pairs": [
      { "from": "idle", "to": "talk", "duration": 0.2 },
      { "from": "talk", "to": "idle", "duration": 0.2 }
    ]
  }
}
```

## 11. AI 模块技术路线（规划，非第一阶段必做）

### 11.1 AI 分割（Segmentation）
目标：从角色图提取 head/torso/arms/legs/hair/eyes/mouth 等 mask。

输出建议格式：
```json
{
  "imageSize": { "w": 2048, "h": 2048 },
  "parts": [
    { "name": "head", "mask": "head.png" },
    { "name": "hair", "mask": "hair.png" },
    { "name": "torso", "mask": "torso.png" }
  ]
}
```

### 11.2 模板匹配（Template Matching）
目标：把分割得到的部件映射到 Schema。

核心问题：
- 部件缺失与合并（例如手臂与衣服粘连）
- 左右判别（left/right）
- 遮挡与层级推断

输出建议：映射关系 + 置信度
```json
{
  "template": "HumanTemplate.v1",
  "mapping": [
    { "part": "head", "source": "mask:head.png", "confidence": 0.92 },
    { "part": "leftArm", "source": "mask:arm_1.png", "confidence": 0.61 }
  ]
}
```

### 11.3 自动 Rig 生成（Rig Generation）
第一阶段不自研骨骼系统，后续可选路线：
- 路线 A：生成 Spine 可用的骨架与权重（难度高、需要完整工具链）
- 路线 B：生成自定义的“部件节点 + 变换约束”模型（轻量，效果较弱）
- 路线 C：半自动：输出可编辑工程（供美术在 Spine 中快速修正）

推荐演进：先走路线 C，确保可控与可产出，再逐步自动化。

## 12. 性能与稳定性方案

### 12.1 帧预算与更新策略
- 目标 60 FPS
- 每帧更新顺序：
  1) Timeline tick
  2) Camera update
  3) Character update（Spine update）
  4) Renderer render

### 12.2 资源与内存
- 背景与角色纹理尽量复用图集
- Scene 切换释放旧 Scene 资源
- 避免频繁创建临时对象（向量/矩阵可复用）

### 12.3 降级策略
- Spine 复杂角色过多时降低：
  - 特效层数量
  - 背景分辨率
  - 非关键动画叠加（如随机 blink 频率）

## 14. 交付物与里程碑（建议）

### 14.0 最小 Demo 资产清单（用于验收 M1）
目标：用最少的资源闭环验证引擎架构，避免“没有资产无法验收”。

必须具备：
- 1 张背景图
  - `assetId`: `bg_classroom_day`
  - 文件：`assets/bg/bg_classroom_day.webp`（格式仅示例，png/jpg 亦可）
- 1 个 Spine 角色 prefab（至少 idle + talk）
  - `prefabId`: `girl_01_spine`
  - 文件集合：`skeleton.json`（或 binary）、`skeleton.atlas`、贴图 png
  - 最低动画要求：`idle`、`talk`（或 mouth track 配置）
- 1 份可运行的 Scene DSL JSON
  - 需要覆盖：SetBackground、Spawn、SetAction、SetEmotion、Talk、Camera.ToShot、Dialogue.Say

建议（用于快速打通链路）：
- 优先使用官方 Runtime 仓库提供的示例角色资源（例如 spineboy）来验证加载、播放与混合逻辑
- 示例资源与 Runtime 同样受 Spine 生态许可约束，使用前需确认授权边界

示例 DSL（可直接用于 M1 验收，概念）：
```json
{
  "version": "1.0",
  "scene": { "id": "classroom_day", "background": { "assetId": "bg_classroom_day" } },
  "characters": [
    { "id": "girl_01", "prefabId": "girl_01_spine", "transform": { "x": 520, "y": 820, "scale": 1.0 }, "state": { "action": "idle", "emotion": "neutral" } }
  ],
  "camera": { "shot": "medium" },
  "timeline": [
    { "t": 0.0, "cmd": "Character.SetEmotion", "args": { "id": "girl_01", "emotion": "happy" } },
    { "t": 0.2, "cmd": "Dialogue.Say", "args": { "id": "girl_01", "text": "你好。" } },
    { "t": 0.2, "cmd": "Character.Talk", "args": { "id": "girl_01", "enabled": true } },
    { "t": 2.0, "cmd": "Character.Talk", "args": { "id": "girl_01", "enabled": false } },
    { "t": 2.2, "cmd": "Camera.ToShot", "args": { "shot": "close", "duration": 0.6 } }
  ]
}
```

### 14.1 里程碑 M1：最小演出闭环
- PixiJS 渲染 + 背景
- Spine 角色加载与显示
- Character.SetAction / SetEmotion / Talk
- Camera shot 切换（wide/medium/close）
- Timeline 执行 Scene DSL 示例

### 14.2 里程碑 M2：可用的剧情控制台
- 支持加载外部 JSON
- 支持运行/暂停/seek
- 支持热切换场景与角色

### 14.3 里程碑 M3：资产规范化与可复用
- 角色 prefab 注册表（prefabId → spine 资源集合）
- emotion map 配置化
- 镜头 shot 配置化

### 14.4 里程碑 M4：半自动 Rig（规划）
- 输出可编辑工程或绑定建议
- 引入模板匹配的置信度与人工修正接口

## 15. 关键风险与对策
- 资源不标准导致表情/动作不可复用：用 prefab 规范 + EmotionMap 数据化兜底
- Spine 资源差异导致混合策略失效：以 MixPolicy 配置覆盖，按角色定制
- Timeline seek 难以准确还原：区分“状态命令”和“一次性命令”，seek 只重放状态命令
- 未来自动 Rig 难度高：先落地半自动（可编辑工程）路径，逐步自动化
